# Fleet Bootstrap

## 1. Purpose

Fleet devices (production Jetsons on aircraft) need a **zero-dependency install path**:
no SSH orchestration, no Bun CLI, no rsync. Just `bash + docker + nvidia-container-toolkit`
(pre-installed at factory).

The one-liner:

```bash
curl -fsSL http://192.168.108.83:8080/install/fastlio | bash
```

With explicit version:

```bash
curl -fsSL http://192.168.108.83:8080/install/fastlio | bash -s -- cuda0.0.0-run-ubuntu20.04-arm64
```

This single command is the entire fleet device deployment surface.

## 2. Architecture

```
Dev-Host (x86_64, 192.168.108.83)
├── Docker Registry (:5000)          ← docker pull
│   └── lio-slam:cuda<tag>-arm64
├── Artifact Server (:8080)          ← curl + wget
│   ├── install/fastlio              bootstrap script
│   └── artifacts/fastlio/           runtime bundles
│       ├── fastlio-runtime-<tag>.tar.gz
│       ├── fastlio-runtime-<tag>.tar.gz.sha256
│       └── latest.txt
└── bun run fleet-bundle             generate + upload

Fleet Device (aarch64, LAN)
├── $HOME/opt/fastlio/
│   ├── etc/device.yaml              hardware config (one-time)
│   ├── runtime/releases/<tag>/      bundle contents
│   ├── runtime/current → releases/<tag>
│   ├── data/PCD/
│   ├── data/logs/
│   └── bin/fastlio-ctl
└── docker run --name fastlio-runtime ...
```

## 3. Bootstrap Script Protocol

The script served at `/install/fastlio` executes the following steps in order.
Any failure before container start reverts the installation.

| Step | Action | Error handling |
| ---- | ------ | -------------- |
| 1 | Read `$HOME/opt/fastlio/etc/device.yaml` → `hardware`, `imu_src` | If missing, print setup instructions, exit 1 |
| 2 | Resolve VERSION (CLI arg or `latest.txt`) | |
| 3 | `wget` runtime bundle `.tar.gz` | Exit 1 |
| 4 | `wget` `.tar.gz.sha256` | Exit 1 |
| 5 | `sha256sum -c` | Delete partial download, exit 1 |
| 6 | `tar xzf` → `$HOME/opt/fastlio/runtime/releases/<version>/` | |
| 7 | `ln -sfn` → `$HOME/opt/fastlio/runtime/current` | |
| 8 | Read `current/manifest.yaml` | Exit 1 |
| 9 | `docker pull <image>` | Exit 1 |
| 10 | `docker stop/rm fastlio-runtime` (if exists) | Ignore errors |
| 11 | `docker run -d ...` (template values from device.yaml) | Exit 1 |
| 12 | Poll `docker exec ... test -f /tmp/slam_ready` (max 30s) | Print warning + tail logs |
| 13 | Print summary (image, version, container ID, log tail) | |

## 4. manifest.yaml Schema

Inside every runtime bundle tarball, at the root level.

```yaml
name: fastlio
version: cuda0.0.0-run-ubuntu20.04-arm64
image: 192.168.108.83:5000/lio-slam:cuda0.0.0-run-ubuntu20.04-arm64
container:
  name: fastlio-runtime
  flags:
    - --network host
    - --ipc host
    - --privileged
  volumes:
    - $HOME/opt/fastlio/runtime/current/config:/catkin_ws/src/bringup/config:ro
    - $HOME/opt/fastlio/runtime/current/launch:/catkin_ws/src/bringup/launch:ro
    - $HOME/opt/fastlio/runtime/current/scripts:/catkin_ws/src/bringup/scripts:ro
    - $HOME/opt/fastlio/data/PCD:/catkin_ws/src/fast_lio/PCD
    - $HOME/opt/fastlio/data/logs:/root/.ros/log
  entrypoint: roslaunch bringup {hardware}_slam.launch imu_src:={imu_src}
entrypoint:
  startup_timeout_sec: 30
  health_poll_interval_sec: 3
```

The bootstrap script substitutes `{hardware}` and `{imu_src}` at runtime from
`$HOME/opt/fastlio/etc/device.yaml`.

## 5. device.yaml Schema

Written once per fleet device by the integrator. Located at `$HOME/opt/fastlio/etc/device.yaml`.

```yaml
hardware: c5v1        # c5v1 | c5pro
imu_src: livox        # livox | mavros
```

If absent, the bootstrap script prints:

```
[fastlio] First-time setup required:
  mkdir -p $HOME/opt/fastlio/etc
  cat > $HOME/opt/fastlio/etc/device.yaml << 'EOF'
  hardware: c5v1
  imu_src: livox
  EOF
```

## 6. Runtime Bundle Contents

Generated by `bun run fleet-bundle` on the dev-host. Tarball name format:

```
fastlio-runtime-<version>.tar.gz
```

Contents:

```
fastlio-runtime-<version>/
├── manifest.yaml
├── launch/                   # bringup/launch/ 镜像
├── config/                   # bringup/config/ 镜像
├── scripts/                  # bringup/scripts/ 镜像
├── rviz_cfg/                 # bringup/rviz_cfg/ 镜像（可选）
└── resource/
    └── MANIFEST              # 仅清单文件，不包含二进制
```

> `resource/` 中的二进制（Livox SDK2、livox_ros_driver2）已内建在 Docker image 中，
> runtime bundle 只带 MANIFEST 用于参考。

## 7. Fleet Device Layout

```
$HOME/opt/fastlio/
├── etc/
│   └── device.yaml               # 硬件配置（人工写入，仅一次）
├── runtime/
│   ├── releases/
│   │   └── cuda0.0.0-run-ubuntu20.04-arm64/
│   │       ├── manifest.yaml
│   │       ├── launch/
│   │       ├── config/
│   │       ├── scripts/
│   │       └── resource/MANIFEST
│   ├── current -> releases/cuda0.0.0-run-ubuntu20.04-arm64
│   └── previous -> releases/<old>     # 回滚时更新
├── data/
│   ├── PCD/                           # 持久化地图
│   ├── bags/                          # 持久化 bag
│   └── logs/                          # ROS + docker logs
└── bin/
    └── fastlio-ctl                    # 本地管理脚本
```

## 8. Rollback

Keep the last 3 releases. Rollback is a manual operation on the fleet device:

```bash
# Roll back to previous release
$HOME/opt/fastlio/bin/fastlio-ctl rollback

# Roll back to specific version
$HOME/opt/fastlio/bin/fastlio-ctl rollback cuda0.0.0-run-ubuntu20.04-arm64
```

Implementation: switch `current` symlink, re-read manifest.yaml, stop/restart container.

The bootstrap script itself performs an **update**, never a rollback. If the new
version fails health check, it prints the logs and exits non-zero, leaving the
previous release intact at `previous` symlink.

## 9. Artifact Server

A standalone HTTP server on the dev-host, port 8080. Implementation: `Bun.serve()`
static file server (`src/web/artifact-server.ts`), managed via:

```bash
bun run fleet-artifacts start      # start server on :8080
bun run fleet-artifacts stop       # stop server
bun run fleet-artifacts status     # show status
```

Filesystem root served: `$HOME/opt/loc-artifacts/` (configurable via `LOCALIZATION_ARTIFACT_DIR` env).

Layout:
```
~home/opt/loc-artifacts/
├── install/
│   └── fastlio                    # bootstrap script (generated from template)
└── artifacts/
    └── fastlio/
        ├── fastlio-runtime-<version>.tar.gz
        ├── fastlio-runtime-<version>.tar.gz.sha256
        └── latest.txt             # plain text, no newline
```

The template lives at `server/artifacts/bootstrap/fastlio.template.sh` in the repo.
`fleet-artifacts start` reads the template, replaces `__ARTIFACT_BASE__` with the
dev-host LAN IP, and writes the result to `install/fastlio` in the serve root.

## 10. Entrypoint Health Signal

The container entrypoint (`docker/entrypoint.sh`) runs a background health poller
whenever the command is `roslaunch bringup ...`. It polls for the `/Odometry` topic
(indicating FAST-LIO is producing output) and writes `/tmp/slam_ready`.

The bootstrap script polls via:

```bash
docker exec fastlio-runtime test -f /tmp/slam_ready
```

## 11. Dev-Host CLI Commands

| Command | Action |
| ------- | ------ |
| `bun run fleet-bundle [version]` | Pack bringup/ → tarball + sha256 → upload to artifact server |
| `bun run fleet-artifacts start` | Start artifact HTTP server |
| `bun run fleet-artifacts stop` | Stop artifact HTTP server |
| `bun run fleet-artifacts status` | Show artifact server status |
| `bun run fleet-artifacts deploy [version]` | Full pipeline: build/push image + fleet-bundle + restage install script |

## 12. CI Integration

After any change to `bringup/` (launch, config, scripts):

```bash
bun run fleet-bundle           # regenerate runtime bundle
bun run fleet-artifacts restart # restart should cause artifact server to re-read files
```

After any C++ change (FAST_LIO, ekf_quat, etc.):

```bash
bun run sync && bun run docker-dbuild && bun run docker-push
bun run fleet-bundle           # update manifest.yaml with current image
```

## 13. Constraints

- Fleet device dependencies: **bash + docker + nvidia-container-toolkit**
  (pre-installed at factory, not installed by bootstrap script).
- Bootstrap script must use **only bash built-ins, curl/wget, sha256sum, tar, docker**.
- No Python, no Node.js, no Bun on fleet devices.
- The bootstrap script is served as `text/plain; charset=utf-8` with shebang.
- Runtime bundles are **immutable** after upload — never overwrite, always new version.
- `latest.txt` is the only mutable reference.

## 14. Version Format

Use the Docker image tag as the bundle version:

```
cuda0.0.0-run-ubuntu20.04-arm64
```

When the tag changes (e.g., `cuda11.4-devel-ubuntu22.04-arm64`), all bundles and
images use the new tag. The tag is the single source of truth for artifact identity.
