# Registry Network Design

## Push and Pull Endpoints

| Direction | From | To | Protocol | Port | NIC |
| --------- | ---- | -- | -------- | ---- | --- |
| Push | dev-device (`192.168.55.1`) | dev-host USB IP | HTTPS (TLS) | `5443` | USB RNDIS |
| Fleet pull | fleet-device | dev-host LAN IP | HTTP | `5000` | `enp6s0` / `wlan0` |
| Tracker upstream | dev-host localhost | `127.0.0.1` | HTTPS (TLS) | `5443` | loopback |

### Rules

1. **Push only via USB RNDIS** (`192.168.55.0/24`). The dev-device has no LAN route to the dev-host.
2. **Fleet pull only via LAN** (`enp6s0` wired or `wlan0` wireless) through the `:5000` HTTP tracker proxy. No TLS on the pull path.
3. **Tracker proxy** runs on dev-host, binds `:5000`, proxies `https://127.0.0.1:5443`. Fleet devices never talk TLS directly.

## Certificate SAN

Generated by `ensureRegistryFiles()` in `src/cli/registry.ts` with 10-year validity. Regenerated automatically on `registry start` when SAN doesn't cover all current IPs.

**Required SAN entries:**

| Entry | Source | Why |
|-------|--------|-----|
| `127.0.0.1` | Always | Tracker upstream link |
| `192.168.55.100` (or actual USB IP) | USB RNDIS | dev-device push target |
| `<LAN_IP>` | `enp6s0` or `wlan0` default route | fleet pull, future direct TLS |

## USB IP Detection (`src/core/network.ts`)

`getAllHostUSBIPs()` uses three-tier fallback to handle the case where the RNDIS link is not yet up when `registry start` runs:

1. **Active route**: `ip route get 192.168.55.1` — works when RNDIS link is up.
2. **Interface scan**: `ip -4 -o addr show` matching `192.168.55.0/24` — works when interface exists even without a default route.
3. **Hardcoded**: `192.168.55.100` — well-known RNDIS gadget host address (always added).

## Pre-Push Validation (`src/cli/docker-push.ts`)

`cmdDockerPush()` validates the cert covers the push target before attempting any docker operation. If the cert SAN doesn't include the push host IP, it errors out immediately:

```
docker-push] Registry cert on dev-host does not cover push target 192.168.55.100.
[docker-push] Run 'bun run registry start' to regenerate the cert, then retry.
```

This avoids the opaque TLS error from the Docker client ("certificate
signed by unknown authority" or "x509: certificate is valid for ... not
...").

## Files

| File | Responsibility |
| ---- | -------------- |
| `src/core/network.ts` | USB/LAN IP detection |
| `src/cli/registry.ts` | Cert generation, SAN coverage check, container lifecycle |
| `src/cli/docker-push.ts` | Push orchestration: trust install, pre-push cert validation |
| `src/core/registry-paths.ts` | Cert and config file paths |
| `src/core/config.ts` | Port numbers, image tag constants |
| `src/web/tracker-server.ts` | HTTP fleet pull proxy |

## Debugging

```bash
# Regenerate cert with current network state
bun run registry stop && bun run registry start

# Verify cert SAN
openssl x509 -in logs/registry/certs/domain.crt -noout -ext subjectAltName

# Retry push
bun run docker-push
```
